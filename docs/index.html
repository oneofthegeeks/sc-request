The Problem

The SC Request Process Has Friction

Today's process works — but it requires too many manual steps and disconnected tools that slow AEs down.

📋

Manual SC Selection

AEs must pick from a full list of SCs every time. Their assigned SC is tracked in a spreadsheet outside of Salesforce.

📅

Disconnected Scheduling

Scheduling with the customer and securing the SC's calendar are two separate steps with no coordination between them.

📧

Unhelpful Notifications

The current email says "See Meeting Invite" instead of the actual time. No direct links to Salesforce records.

📊

No Reporting History

SC requests are only tracked as Events. There's no structured record to report on SC utilization or request patterns.

Proposed Solution

A Smarter SC Request — Without Leaving Salesforce

The same "Request SC" button AEs already know. A completely redesigned experience behind it.

🎯

Auto-Assigned SC

The AE's assigned SC is pre-populated automatically. Override available if needed. No more hunting through a full list.

📆

Live Calendar Availability

See the next 5 open slots for both AE and SC in real time. Pick a time while on the phone with the customer.

📨

Instant Outlook Invite

Optionally send a calendar invite to both AE and SC automatically on submit. The invite body contains all the same details as the email notification — no separate step required.

📈

Full Request History

Every SC request is stored as a structured record on the Opportunity — reportable, searchable, always accessible.

Zero Retraining Required
The entry point stays exactly the same — "Request SC" in the Opportunity action bar. AEs don't need to learn anything new about where to find it.
Request Flow

How It Works — Step by Step

The AE completes the entire process in one modal. Everything else happens automatically.

1

AE clicks "Request SC" in the Opportunity action bar

Same button, same location. A Lightning modal opens instantly — no page navigation required.

2

SC is auto-populated from the AE's assignment

The system looks up the AE's assigned SC automatically. The AE can override and choose any SC from a dropdown if needed.

3

Live calendar availability loads

The next 5 open time slots for both the AE and SC appear instantly. The AE can also browse a specific date. Duration defaults to 1 hour.

4

AE selects request type, product line, and fills in details

Dropdowns for Request Type (14 options) and Product Line (17 options). Freeform notes field for context, current state, and demo requirements.

5

AE reviews and submits

Optional checkbox: "Send Outlook calendar invite to SC automatically" — checked by default. The invite includes full request details (opportunity, contact, type, notes, SC) matching the email notification. One click to submit.

✓

Everything else happens automatically

SC Request record created  ·  Event logged on Opportunity  ·  SC added to Opportunity Team  ·  Opportunity fields updated  ·  Email sent to AE, SC, and both managers  ·  Outlook invite sent (if selected)

Calendar Integration

Schedule While You're on the Phone

The AE sees real availability for both themselves and the SC — no back-and-forth, no separate scheduling step.

⚡

Next 5 Open Slots

The system shows the next 5 times that work for both AE and SC automatically — no date hunting required.

🗓️

Date Picker

If the AE has a specific date in mind from the customer conversation, they can select it and see all open times that day.

⏱️

Duration Options

30 min, 1 hour, or 90 min. Defaults to 1 hour. Slots update instantly when duration changes.

📬

Optional Outlook Invite

After picking a slot, the AE can auto-send an Outlook invite to both AE and SC. The invite body includes all request details — opportunity, contact, type, product, notes, and a link back to Salesforce. Opt-out if already scheduled.

If calendar is unavailable
The form never blocks submission. If the Microsoft 365 API is unavailable or no slots are found, the AE can always enter a date and time manually and proceed.

Implementation Options

Two Ways to Build It

Both options deliver the same experience and write the same data to Salesforce. The difference is the platform.

Criterion Option A — Salesforce Native Option C — Slack Workflow
Where AE works Inside Salesforce — no context switch Slack — separate from SFDC
Opportunity context Automatic — already on the record AE must search for Opportunity
Build complexity Higher — LWC + Apex + Connected App Medium — Node.js + Slack Bolt SDK
Iteration speed Slow — Salesforce deployment process Fast — redeploy backend service
IT/Security compliance Native SFDC — already approved New Slack app — needs approval
Writes to Salesforce Yes — natively Yes — via REST API
Long-term maintenance One system to maintain SFDC + Slack app + backend service
Recommendation: Option A — Salesforce Native
AEs already live in Salesforce. Keeping everything native means no context-switching, no extra systems to maintain, and no new IT approvals. The higher build complexity is a one-time cost — the long-term simplicity is worth it.

Technical Reference

SC Selection Logic

On modal load, the LWC queries SC_Pairing__c for the current user and handles four scenarios.

Scenario 1 — Normal

Exactly one active pairing found

SC field is auto-populated with the paired SC. AE can still override via dropdown at any time. No warning shown.

Scenario 2 — Data Integrity Issue

Multiple active pairings found for this AE

Use the most recently created record (CreatedDate DESC). Show an inline warning: "Multiple active SC pairings found — please verify the SC selected is correct." Automatically email the SC's manager (SC.ManagerId) to clean up the duplicate records.

Scenario 3 — No Pairing

No active pairing found

SC field defaults to a full dropdown of all active SCs with no pre-selection. AE must manually choose.

Scenario 4 — Override

AE overrides the auto-populated SC

Available at any time regardless of which scenario applied. AE selects any active SC from the dropdown. Calendar availability re-queries automatically for the newly selected SC.

Important — One SC can have many AEs
One SC being paired to multiple AEs is normal and expected. The constraint is one-directional: one AE should not have more than one active SC pairing at the same time. The validation rule on SC_Pairing__c enforces this.

SOQL Query

SELECT SC__c, SC__r.Name, SC__r.ManagerId, CreatedDate FROM SC_Pairing__c WHERE AE__c = :UserInfo.getUserId() AND Active__c = true ORDER BY CreatedDate DESC LIMIT 2 -- LIMIT 2 lets us detect the duplicate case without fetching all

Validation Rule — One Active Pairing Per AE

/* SC_Pairing__c — One_Active_Pairing_Per_AE */ AND( Active__c = TRUE, VLOOKUP( $ObjectType.SC_Pairing__c.Fields.Id, $ObjectType.SC_Pairing__c.Fields.AE__c, AE__c ) <> Id ) Error: "This AE already has an active SC pairing. Deactivate the existing pairing before creating a new one."

M365 Calendar Integration

The integration uses Microsoft Graph API free/busy endpoint via a Salesforce Named Credential with per-user OAuth.

API Endpoints Used

EndpointMethodPurpose
https://graph.microsoft.com/v1.0/me/calendarView/getSchedulePOSTQuery free/busy for AE + SC simultaneously
https://graph.microsoft.com/v1.0/me/eventsPOSTCreate Outlook calendar event (when invite checkbox is checked)

OAuth Scopes Required

Calendars.Read — free/busy query Calendars.ReadWrite — create Outlook invite

IT team to confirm whether the existing SSO / App Registration can be extended to cover these scopes, or whether a new App Registration is needed. This is the one open question remaining before development can begin.

SettingValue
LabelM365 Graph API

Per-User Auth Onboarding
Because the Named Credential uses NamedUser principal type, each Salesforce user must authenticate with their Microsoft account once before they can use the calendar feature. The LWC detects if authentication is missing and prompts the user to complete the OAuth flow. This is a one-time step per user.

Outlook Invite Behavior

StateBehavior
Checkbox defaultChecked (opt-out model). Most requests won't have a pre-existing invite.
If checked on submitM365 integration creates an Outlook calendar event for both AE and SC. The invite body matches the email notification: Opportunity name (linked), Account, Primary Contact, Meeting Date & Time, Request Type, Product Line, Assigned SC, AE notes, and a "View Opportunity" button. SC receives the invite in Outlook to accept or decline.
If uncheckedNo Outlook invite sent. AE handles calendar coordination. Salesforce Event record is still created regardless.

Error & Fallback Handling

M365 API unavailable

Calendar availability is currently unavailable

Inline error shown. AE can still manually enter a date and time and proceed with submission. Form is never blocked.

Zero open slots in next 5 business days

"No mutual availability found in the next 5 days"

Prompt shown. AE can use the date picker to check specific dates, or enter a time manually.

Manual entry fallback

Always available regardless of API status

A manual date/time input is always present in the form as a fallback. No dependency on M365 availability to complete a request.
