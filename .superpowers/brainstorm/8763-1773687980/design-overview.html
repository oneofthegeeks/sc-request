SC Request System — Full Design

Two options fully specced. Option A is the recommended primary build.

Shared Foundation (Both Options)

Salesforce Objects

  • SC_Request__c — new custom object, full request record
  • Event — created on submit, title format preserved
  • OpportunityTeamMember — SC added as Solution Consultant
  • User — AE, SC, managers via ManagerId
  • SC_Pairing__c — new custom object, AE→SC mapping (imported from spreadsheet)

Opportunity Fields Updated on Submit

  • Primary Solution Consultant — auto-populated
  • SC Date Requested — auto-populated
  • SC On Team ✓ — auto-checked
  • Demo ✓ — auto-checked if request type = Demo
  • SC Request Notes — NEW field, AE's freeform details

Static Picklists (dev-ticket to update)

  • Request Type: Demo, Discovery, + others
  • Product Line: GoTo Connect, Resolve, + others

Notifications on Submit

  • Email → AE
  • Email → SC
  • Email → AE's Manager (via User.ManagerId)
  • Email → SC's Manager (via User.ManagerId)
  • All emails include full request details

Calendar Availability (M365 Integration)

Queries Microsoft 365 free/busy for both AE and SC simultaneously. Shows:
  • Quick slots: Next 5 open windows that work for both (default view)
  • Date picker: AE can select a specific date to see all open times that day
  • Duration options: 30 min / 1 hour / 90 min
  • Business hours only (configurable)

Option A — Salesforce Native (Recommended)

How it works

A Lightning Web Component (LWC) button — "Request SC" — placed in the Sales Engineering Info section of the Opportunity layout. Clicking it opens a Lightning modal with a multi-step form. On submit, an Apex controller handles all writes (Event, OpportunityTeamMember, SC_Request__c, Opportunity field updates). Email alerts fired via SFDC Email Alert automation. M365 calendar queried via a Named Credential + External Service (OAuth 2.0 Connected App).

SC Pairing management: SC managers maintain pairings directly in a simple Salesforce list view on the SC_Pairing__c object — no spreadsheet needed after initial import. Changes take effect immediately.

Build complexity: High — requires LWC, Apex, Connected App OAuth setup, Named Credentials
Change velocity: Slow — Salesforce deployment process for any UI changes
User experience: Best — AE never leaves Salesforce

Option C — Slack Workflow (Alternative)

How it works

A Slack app with a /request-sc slash command. Opens a Slack modal with the same form fields. AE searches for the Opportunity by name (type-ahead via Salesforce API). On submit, a Node.js/Python backend service handles all Salesforce writes (same objects as Option A) and M365 calendar queries. Calendar availability posted as interactive Slack message buttons. Confirmation DM sent to AE and SC in Slack, plus the same 4-recipient email.

Key difference: Opportunity lookup adds one extra step vs. Option A, but the workflow can be triggered from anywhere in Slack — ideal for AEs who work primarily in Slack.

Build complexity: Medium — Slack app + backend service + Salesforce API + M365 API
Change velocity: Fast — redeploy backend, no Salesforce deployment needed
User experience: Good — native Slack feel, but context switch from SFDC

Event Title Format (Preserved)

"Request for [Request Type] for [Account Name] w/ a Solutions Consultant"
Event description = SC Request Notes (AE's freeform details)
