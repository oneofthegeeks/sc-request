Three Approaches for SC Request

Each delivers the same core workflow — pick the platform that fits your team best

A

Salesforce Native

Custom Lightning component on the Opportunity record. AE never leaves SFDC. Uses SFDC flows, Apex, and M365 calendar API integration.

Pros

  • Lives where AEs already work
  • Directly tied to the Opportunity record
  • SFDC data (managers, teams) auto-available
  • No separate login or tool

Cons

  • Slower to build and change
  • Requires Salesforce developer expertise
  • M365 OAuth adds complexity inside SFDC
  • Harder to iterate on UX
B

Standalone Web App

A purpose-built web app (e.g., Next.js) that integrates with Salesforce via API and M365 for calendar. AE opens it from a link on the Opportunity or from a bookmark.

Pros

  • Fast to build and easy to iterate
  • Full control over UX and calendar UI
  • Can embed in SFDC as a tab/iframe
  • No SFDC deployment process

Cons

  • AE leaves Salesforce to use it
  • Separate auth/login to manage
  • Needs SFDC + M365 API integrations
C

Slack Workflow ⭐ Recommended

A Slack workflow app. AE triggers it from any Slack channel or DM. Multi-step modal collects all request info, checks M365 calendar availability, posts a confirmation, and writes back to Salesforce via API.

Pros

  • AEs already live in Slack
  • Fastest interaction — modal feels native
  • SC gets notified in Slack instantly
  • Easy to update and redeploy
  • Can still write everything back to SFDC
  • Slash command: /request-sc

Cons

  • Requires Slack app infrastructure
  • AE must paste/search Opportunity ID or name
  • Less visual than a web UI for calendar picker

My Recommendation

Option C (Slack) as the primary interface, with Option B (Web App) as the calendar picker — the Slack modal handles the request form, but when the AE needs to pick a time slot, it links out to a lightweight web page showing availability. This gives you the best of both worlds: Slack speed + rich calendar UX. Both write back to Salesforce automatically.
